Contributing

The lay of the land: how the repository is structured, the decisions and principles behind it, the workflow the history was built with — and the recipe for adding a feature.

Repository tour

Two structural facts do a lot of work. Everything below main.rs lives in the library target, so unit tests, e2e tests and the property suites exercise the exact code the binary runs. And all internal offsets are byte offsets into the document text — conversion to LSP positions happens in exactly one place (position.rs), parametrized by the encoding negotiated with the client. If you find yourself converting positions anywhere else, stop.

Adding a feature

The extensibility rule: feature N+1 = one new file in src/features/ + one line in Registry::standard(). A capability tweak in capabilities.rs is needed only when a feature introduces a new LSP method or action kind — rare; most features are code actions of an already-advertised kind.

Worked sketch — Unquote cell, the first backlog item. Test-driven, like everything here:

  1. Red. Create src/features/unquote.rs with a failing test using the shared helpers: 
    #[cfg(test)]
mod tests {
    use super::*;
    use crate::features::testutil::{ctx_at, doc};

    #[test]
    fn unquotes_a_quoted_cell_whose_value_survives_bare() {
        let doc = doc("a,\"b\"\n");
        let actions = UnquoteCells.actions(&ctx_at(&doc, doc.text.find('b').unwrap()));
        assert_eq!(actions[0].title, "Unquote cell");
        assert_eq!(crate::edits::apply(&doc.text, &actions[0].edits), "a,b\n");
    }

    #[test]
    fn cells_that_need_their_quotes_offer_nothing() {
        let doc = doc("a,\"b,c\"\n"); // the comma needs protection
        assert_eq!(UnquoteCells.actions(&ctx_at(&doc, 3)), Vec::new());
    }
}
  2. Green. Implement the provider. Applicability is recomputed from the parsed table — never from client-echoed diagnostics — and the edit replaces the cell’s content span, so alignment padding survives: 
    pub struct UnquoteCells;

impl ActionProvider for UnquoteCells {
    fn name(&self) -> &'static str {
        "unquote-cells"
    }

    fn actions(&self, ctx: &ActionContext) -> Vec<Action> {
        let Some((row, column)) = ctx.cell_at_cursor() else {
            return Vec::new();
        };
        // qualify: Quoted, on a clean row, and the decoded value contains
        // no delimiter/quote/newline (encode_cell would re-quote it).
        // edit: replace cell.content_span with the bare value.
        // kind: REFACTOR_REWRITE — the inverse of quote.rs.
        …
    }
}
  3. Register. One line in features/mod.rs: providers: vec![…, Box::new(unquote::UnquoteCells)] — plus the pub mod unquote; declaration. Nothing else: the server’s code-action handler, the kind filter, and the action-contract property pick the provider up automatically through the registry.
  4. Prove it end-to-end. Add an e2e scenario (open → action → apply → didChange → clean diagnostics) in tests/e2e.rs, and run PROPTEST_CASES=10000 cargo test once — the contract property will apply your edits to thousands of generated documents and re-parse the result.
  5. Gate and commit (workflow below): one commit for the cycle, e.g. feat(features): unquote cell where the value survives bare.

What providers may rely on: the parser invariants, the three ground rules (header contract, verbatim error rows, padding is layout), and ActionContext’s cursor helpers. What they must uphold: edits non-overlapping and in document order, no no-op actions (return nothing instead), and applicability recomputed from the table.

Design decisions

The four load-bearing choices, in the order they were made (originally maintained as ADR files; consolidated here — commit messages referencing “ADR 000N” mean these).

1 · Rust rather than Zig 2026-07-02

Both produce small, fast, single-binary language servers. The server’s value lies in CSV-specific features, not protocol plumbing, so ecosystem maturity decided: lsp-server/lsp-types are maintained under rust-lang and power rust-analyzer, while Zig’s LSP tooling tracks a pre-1.0 toolchain whose std breaks between releases — a recurring maintenance tax — and the JSON-RPC loop would be hand-rolled. Helix itself is Rust, so the contributor pool overlaps the target audience. Accepted costs: compile times, the borrow checker’s learning curve. Zig’s genuine strengths (compile speed, cross-compilation) were parity or not decisive for a long-lived tool.

2 · lsp-server + lsp-types, synchronous — not an async framework, not hand-rolled 2026-07-02

Hand-rolling the transport is easy (~200 lines) but the LSP type surface is enormous and versioned — reimplementing it invites subtle bugs and buys nothing. Async frameworks (tower-lsp, unmaintained for years; its forks; async-lsp) add a tokio runtime and service abstractions to a server whose every request is answerable in microseconds: pure overhead. So: the synchronous scaffold rust-analyzer is built on — a plain channel of messages consumed in a for loop, one thread, no tokio in the tree. Long-running requests would block the loop — acceptable by design, nothing here is long-running. lsp-types is pinned at the minor version (0.97 switched Url → its own Uri), and URIs are treated as opaque keys throughout to stay insulated from that churn.

3 · Hand-written error-tolerant parser, not the csv crate 2026-07-02

An IDE backend needs per-cell spans, error reporting with recovery, and a lossless layout model; the csv crate is a lenient data reader that silently repairs malformed input and discards layout — destroying exactly the information a language server must surface. The parser page has the full comparison. Consequences embraced: the parser is the project’s core asset with the deepest test coverage, and every RFC 4180 edge-case decision is owned and documented here.

4 · Property-based testing with proptest 2026-07-03

The robustness story rests on documented invariants, and hand-picked examples only cover known failure modes. proptest is the closest Rust equivalent of Hypothesis: strategy combinators, integrated shrinking, failure persistence that yields a growing committed regression corpus. Chosen over quickcheck (type-directed shrinking, poor for fragment-built strings), bolero (indirection not yet justified) and cargo-fuzz (nightly, indefinite runtime — the documented escalation path, not a CI gate). Dev-dependency only; the shipped binary is unaffected. Details on the testing page.

Design principles

  • Byte spans everywhere, one conversion boundary. Features never see LSP positions; position.rs owns the encoding mess.
  • The parser is total. Malformed input produces a Table plus errors — never a failure. Everything downstream may assume a table exists.
  • The verbatim principle. Rows the parser could not fully understand are excluded from column checks and pass through every rewrite byte-for-byte.
  • Padding is layout, not data. Values are trimmed content; align ⇄ compact round-trips byte-identically; quoted interiors are untouchable.
  • Recompute, never trust. Client-echoed diagnostics and data payloads are response decoration; applicability always comes from the freshly parsed table.
  • Edits are eager, minimal, and data. Complete WorkspaceEdits (no resolve round trip), shrunk to one small TextEdit so cursors stay put and formatting is idempotent.
  • Requests are always answered; notifications never are. Panics are contained per-request by catch_unwind; malformed notifications are logged and dropped; unknown documents are tolerated.
  • stdout is sacred. Protocol frames only; logs to stderr behind CSV_LSP_LOG=1.
  • No configuration surface (yet). Sane behavior over knobs until real usage demands otherwise (see backlog).

Workflow

Test-driven, in small cycles, one commit per cycle:

  1. Red — write the smallest test that fails for the right reason. Run it, watch it fail. If it passes immediately, the test is wrong or the step is already done.
  2. Green — the minimal implementation that passes. Resist generalizing beyond the test.
  3. Refactor — with tests green, clean up naming and duplication. No behavior change.
  4. Gate — all four must pass locally before every commit: 
    cargo fmt --all
cargo clippy --all-targets -- -D warnings
cargo test
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps
  5. Commit — test + implementation together (history stays bisectable; never commit a red state).

Commits follow Conventional Commits: <type>(<scope>): <imperative summary ≤ 72 chars>, body explaining the why when it isn’t obvious. Types used here: feat, fix, test, refactor, docs, chore, ci; the scope is the module (parse, position, server, features, …). Examples from the history:

feat(position): map byte offsets to utf-16 positions
test(parse): cover unclosed-quote recovery
fix(edits): keep minimized edit boundaries off CRLF middles
docs: document property testing and refined invariants

CI (.github/workflows/ci.yml) runs the same four gates on stable Rust with Swatinem/rust-cache, on pushes to main and on PRs. CI is the enforcement mechanism; the local loop is the fast path.

Code style and error handling

  • rustfmt with the default configuration — no rustfmt.toml; the community default is the convention. clippy clean at -D warnings.
  • Lint configuration lives in Cargo.toml, not in source attributes: 
    [lints.rust]
unsafe_code = "forbid"    # this project has no reason to ever use unsafe
missing_docs = "warn"     # every public item carries a doc comment; CI escalates via -D warnings
  • Doc comments state invariants (“spans lie on char boundaries”), not restatements of the signature; every module opens with a //! line naming its single responsibility.
  • unwrap()/expect() are acceptable in tests and in main.rs startup, nowhere else. Library code returns the error or handles it locally; the dispatch layer maps errors to LSP error responses, with catch_unwind as the last-resort belt.
Rust concept · a five-line logging framework

The only logging infrastructure is one declarative macro in lib.rs:

#[macro_export]
macro_rules! log {
    ($($arg:tt)*) => {
        if std::env::var_os("CSV_LSP_LOG").is_some() {
            eprintln!("[csv-lsp] {}", format!($($arg)*));
        }
    };
}

$($arg:tt)* captures any token stream and forwards it to format!, so log!("initialized with {encoding:?}") works like println! — but to stderr, and only when the environment variable is set. #[macro_export] hoists it to the crate root. When a dependency would be one macro long, write the macro.

Dependency policy

Four runtime dependencies — lsp-server, lsp-types (minor-pinned), serde_json, unicode-width — plus proptest as a dev-dependency. Adding one requires a reason worth writing down in Design decisions. Cargo.lock is committed: this is a binary crate, and reproducible builds trump lockfile churn.

Performance stance

Full reparse on every keystroke is O(n) and fine into the tens of megabytes; FULL document sync keeps the server trivial. The isolated escape hatches, if ever needed: incremental sync (contained in Document) and a size threshold for align. Do not optimize before that shows up in practice.

Backlog

All features from the original scope are implemented. Candidates for later, roughly in order of appetite:

  • Unquote cell/column — the inverse of the quote actions, only where the value survives unquoted (the worked example above).
  • Extra-cells quickfix — merging or dropping surplus cells; deleting data needs more care than adding empty cells.
  • Configuration via initializationOptions — severity overrides, blank-line policy, header conventions.
  • Incremental sync if very large files ever make FULL sync noticeable.
  • Rename-file-on-convert via LSP RenameFile resource operations, where clients support them.

Maintaining these docs

  • Each page in docs/ is deliberately self-contained (inline CSS/JS, no build step, no network). The style and script blocks are duplicated verbatim across pages — when touching them, apply the change to every page.
  • Three artifacts must stay in sync when behavior changes: the prose invariant (parser page), the property that checks it (invariant map), and the code. A change to one is a change to all three.
  • Feature GIFs are reproducible artifacts: tapes and samples live in docs/gifs/, outputs are recorded on demand (recording guide).
  • Code comments reference these pages by path (e.g. docs/parser.html) — keep anchors stable when restructuring.
