Testing

How this project stays correct: a small pyramid of example-based tests, and a property-based layer that hunts for the inputs nobody thought to write down.

Philosophy

The codebase was built test-driven in small red → green → refactor cycles, one commit per cycle — so the tests are not an afterthought, they are the executable form of every behavioral decision documented in these pages. Two consequences shape everything below:

  • Tests assert behavior, not implementation. Names describe facts (padding_is_trimmed_from_content_spans, not test_parse_2), and parser tests compare span slices — assert_eq!(span.slice(text), "…") — so failures read like text, not index arithmetic.
  • Documented invariants get machine-checked counterparts. Every guarantee in the invariant catalog maps to a property test (the map).

Four gates must pass before every commit — CI (.github/workflows/ci.yml) runs the identical four on stable Rust, so green locally means green remotely:

cargo fmt --all
cargo clippy --all-targets -- -D warnings
cargo test
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps

The e2e layer deserves two notes, because its conventions keep CI honest:

  • Every receive uses recv_timeout, never a blocking recv — a hung server fails the test in five seconds instead of hanging the CI job.
  • The TestClient helper plays the editor faithfully: it buffers notifications that arrive while waiting for a response (a command republishes diagnostics before answering), and it applies TextEdits client-side — back-to-front, in the negotiated encoding — exactly as an editor would, then syncs the result back with didChange. Feature tests therefore exercise the entire loop: diagnostics → action → edit → reparse → clean diagnostics.
Rust concept · test targets

Files in tests/ are integration tests: each is compiled as its own crate that can only see the library’s public API — which is exactly the point, and why everything below main.rs lives in lib.rs. The stdio test finds the real binary through env!("CARGO_BIN_EXE_csv-lsp") — cargo builds the binary target and injects its path at compile time. And in-module #[cfg(test)] unit tests compile away entirely in release builds.

Property-based testing

Hand-picked examples only cover the failure modes someone already imagined. The interesting bugs live in the inputs nobody wrote down — and an LSP server has two untrusted input surfaces: document text (anything a user opens) and protocol traffic (anything a client sends). Property-based testing covers both: describe how to generate inputs, state a property that must hold for all of them, and let the framework hunt.

The tool is proptest — the closest Rust relative of Python’s Hypothesis, chosen over the alternatives deliberately:

ToolModelWhy (not) here

Strategies: generating CSV-shaped trouble

A Strategy is a composable recipe for generating (and shrinking) values. The document suite’s core insight is that uniform random strings are useless here: a random Unicode string almost never contains a quote, so it almost never reaches the parser’s deep states (escapes, recovery, multi-line cells). Instead, documents are assembled from a small vocabulary of the parser’s interesting bytes, weighted by repetition:

/// Weighted-by-repetition vocabulary: uniform random strings almost never
/// contain a quote or a delimiter, so fragment assembly is what reaches the
/// deep parser states (quoting, escapes, recovery).
static FRAGMENTS: &[&str] = &[
    ",", ",", ",", ";", ";", "\t", "\t", "|", "|", // delimiters of all dialects
    "\"", "\"", "\"", "\"\"", // quotes and escaped quotes
    "\n", "\n", "\r\n", "\r", // all three terminators
    " ", " ", "\u{feff}", // padding and the BOM (also mid-text)
    "a", "a", "b", "x9", "é", "😀", "名", "\0", // content, multibyte included
];

fn csv_shaped() -> impl Strategy<Value = String> {
    prop::collection::vec(prop::sample::select(FRAGMENTS), 0..64)
        .prop_map(|parts| parts.concat())
}

/// Mostly CSV-shaped, sometimes arbitrary Unicode noise.
fn wild_text() -> impl Strategy<Value = String> {
    prop_oneof![
        4 => csv_shaped(),
        1 => any::<String>(),
    ]
}

Three combinators do all the work: select picks from a slice, collection::vec repeats a strategy, prop_map transforms values, and prop_oneof! mixes strategies with weights. The same style builds protocol traffic in tests/protocol.rs: coordinates are “mostly small, sometimes 2000, sometimes u32::MAX”, JSON params are recursively generated junk.

One pattern deserves a name, because it avoids proptest’s biggest trap. Some properties need clean documents (dialect conversion is only offered on files without quoting errors). Filtering generated inputs (prop_filter) would reject most cases and blow the rejection budget. Instead the suite repairs constructively:

/// A document that parses without errors under the paired dialect. Every
/// parse error requires a quote byte, so stripping the quotes repairs any
/// broken input constructively — no rejection budget, any case count.
fn clean_text_and_dialect() -> impl Strategy<Value = (String, Dialect)> {
    (wild_text(), dialect()).prop_map(|(text, from)| {
        if parse(&text, from).errors.is_empty() {
            (text, from)
        } else {
            (text.replace('"', ""), from)
        }
    })
}
Rust concept · strategies are values

impl Strategy<Value = String> is an ordinary return type — strategies are plain values built by chaining combinators, so they can be named, reused and composed like iterators. Inside the proptest! { … } macro, text in wild_text() binds each generated value; prop_assert!/prop_assert_eq! report failures back to the runner (so it can shrink) instead of panicking outright — use them instead of assert! inside properties. Helper functions called from a property may still use plain assert!; a panic is also treated as a failure, it just skips the nicer reporting.

Shrinking, seeds, and the regression corpus

When a case fails, proptest shrinks it: it re-runs the property on systematically smaller variants (fewer fragments, simpler characters) until no smaller input still fails, then reports that minimal counterexample. A 60-fragment document with a bug usually shrinks to something like "\r — small enough to reason about directly.

The failing seed is then written next to the suite — tests/properties.proptest-regressions / tests/protocol.proptest-regressions — and those files are committed: every future run replays past failures first, which is the Hypothesis example-database workflow, and CI inherits the whole bug history for free. (The suites configure FileFailurePersistence::WithSource explicitly, because proptest’s default location probes for lib.rs/main.rs and silently gives up inside integration-test targets.)

Case counts are tuned so cargo test stays in seconds: the document suite runs the default 256 cases per property; the protocol suite runs 64 richer cases (each spawns a real server thread). Deepen locally whenever you touch the parser or renderer:

PROPTEST_CASES=10000 cargo test --test properties
PROPTEST_CASES=1000  cargo test --test protocol

Note the pattern in the big action property: it does not just check that actions look right — it applies their edits and re-parses the result, asserting the outcome is structurally sound under the dialect the document ends up in (conversions included). Properties compose: assert_table_sound written once for the parser gets reused as the postcondition of every feature.

The hostile-client suite

tests/protocol.rs generates entire LSP sessions against the real server over Connection::memory(): initialize with possibly-bogus encoding offers ("wtf-8"…), then a random sequence of operations —

  • opens, changes and closes in any order, including changes to never-opened documents and a ranged change under FULL sync (a client bug the server shrugs off);
  • code-action requests with reversed and absurd ranges (u32::MAX) and random kind filters; formatting and highlight requests anywhere;
  • executeCommand with wrong names, wrong arity, junk arguments (["file:///w/a.csv", "klingon"]);
  • raw requests and notifications with arbitrary JSON params, unknown methods included.

The property: every request is answered (a five-second timeout fails the test — no hangs), no response ever comes from the panic backstop (an error message containing “panicked” fails the run — catch_unwind is a belt, not a license), every pushed notification is well-formed diagnostics with ordered ranges, and the session ends in a clean shutdown with the server thread joining Ok.

Running and extending

cargo test                                  # everything: unit + golden + corpus + e2e + properties
cargo test --test properties                # one suite
cargo test padding_is_trimmed               # one test by substring
PROPTEST_CASES=10000 cargo test --test properties   # deeper hunt (do this after parser/render changes)

When a property fails, the workflow is:

  1. Read the shrunk counterexample in the failure message — it is usually two or three characters. Reproduce it in a scratch unit test if the property body is broad.
  2. Fix the code (or, rarely, the property — if the property was wrong, the documented invariant it mirrors needs the same correction; keep prose and property in sync).
  3. Commit the updated tests/*.proptest-regressions file together with the fix. That seed is now replayed first on every machine, forever.
  4. Consider whether the minimal case deserves promotion to a named example-based test — regressions with a story are better documentation than seed hashes.

When adding a feature, you usually need no new property: the action contract property picks up new providers automatically through Registry::standard() — one more reason the registry is the single registration point. Add a dedicated property only for feature-specific semantics (as pad_fix_all_repairs_every_short_row does), and add e2e coverage for the editor-visible flow.
