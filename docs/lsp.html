LSP setup in this project

How the Language Server Protocol works, how csv-lsp is wired into it, and where every message goes — from the stdio pipe to the parsed table and back.

The protocol in ten minutes

An LSP server is a process that speaks JSON-RPC 2.0 over stdio. Every message is a JSON object prefixed with a length header:

Content-Length: 85\r\n
\r\n
{"jsonrpc":"2.0","id":1,"method":"initialize","params":{...}}

Two message flavors, and the distinction drives the whole server design:

  • Request — has an id; the receiver must answer with a response (result or error) carrying the same id.
  • Notification — no id; never answered, not even with an error.

A session (client = editor, server = csv-lsp) has a fixed lifecycle:

  1. Client sends initialize (request) with its capabilities; the server answers with its own — the contract of what may be called later.
  2. Client sends initialized (notification). The session is live.
  3. Document sync flows as notifications: textDocument/didOpen (full text + languageId + version), didChange, didClose. Feature traffic flows as requests: codeAction, formatting, documentHighlight, executeCommand.
  4. The server pushes textDocument/publishDiagnostics notifications whenever it likes — csv-lsp: after every open and every change. Each publish replaces the previous set for that file.
  5. Client sends shutdown (request), then exit (notification); the process ends.

The crates, and why the server is synchronous

csv-lsp is built on lsp-server + lsp-types — the scaffold rust-analyzer uses — rather than an async framework or hand-rolled plumbing. The reasoning (decision record, condensed):

  • Transport is trivial (~200 lines if hand-rolled), but the LSP type surface is enormous — hundreds of carefully versioned structs. lsp-types provides all of them spec-correct and serde-ready.
  • CSV parsing is O(n) and takes microseconds to milliseconds; every request can be answered synchronously. A tokio runtime plus service layers would be pure overhead. rust-analyzer proves the synchronous architecture at scales far beyond this project’s.
  • lsp-server contributes exactly the boring parts: Connection with stdio() and memory() constructors (the latter powers the e2e tests), Content-Length framing, initialize_start()/initialize_finish() handshake helpers, and handle_shutdown().

The whole server is one thread plus lsp-server’s two I/O threads; main.rs is a 20-line shim that connects stdio, runs the loop, and joins the I/O threads on the way out. The main loop consumes a plain channel:

for message in &connection.receiver {
    match message {
        Message::Request(request) => {
            if connection.handle_shutdown(&request)? {
                return Ok(());
            }
            let response = handle_request(&mut state, &connection, request);
            connection.sender.send(Message::Response(response))?;
        }
        Message::Notification(notification) => {
            handle_notification(&mut state, &connection, notification)?;
        }
        // We never send server→client requests, so no responses arrive.
        Message::Response(_) => {}
    }
}
Rust concept · traits as protocol tables

lsp-types models every protocol method as a type implementing lsp_types::request::Request (or …::notification::Notification) with two associated items: the method name constant and the params type. Dispatch never touches string literals or untyped JSON twice:

fn cast_request<R: lsp_types::request::Request>(
    request: Request,
) -> Result<(RequestId, R::Params), RequestError> {
    request.extract(R::METHOD).map_err(|err| RequestError {
        code: ErrorCode::InvalidParams,
        message: format!("invalid `{}` params: {err}", R::METHOD),
    })
}
// call site — the turbofish picks the method:
let (id, params) = cast_request::<CodeActionRequest>(request)?;

One generic function replaces a family of hand-written deserializers, and the compiler guarantees the method string and the params type can never disagree.

Dataflow

The design invariant that keeps this picture simple: everything inside the process works in byte offsets into the document text. LSP positions exist only at the boundary — position.rs converts incoming positions to offsets before features run, and outgoing spans to positions after they are done. No other module is allowed to do this conversion.

Walkthrough: a keystroke

  1. The editor sends didChange. csv-lsp advertises FULL sync, so the notification carries the complete new text — no delta bookkeeping, no chance of drifting out of sync. (Reparsing is O(n) and fine into the tens of MB; incremental sync is the documented escape hatch if that ever changes.)
  2. Store::change replaces the text and eagerly rebuilds the derived state: the LineIndex and the parsed Table. The dialect is not re-detected — it was fixed at open so a half-typed document cannot flip dialect under your cursor.
  3. publish_diagnostics runs every registered rule over the table, converts each internal Diag (byte span, severity, code) into an LSP diagnostic at the boundary, and pushes one publishDiagnostics notification tagged with the document version.

Walkthrough: a code action

  1. The editor asks: textDocument/codeAction with a range (a bare cursor is a zero-width range) and a context — its current diagnostics for the range, plus an optional only kind filter.
  2. The server converts both positions to byte offsets — normalizing reversed ranges from hostile clients (start.min(end)…) — and builds an ActionContext.
  3. Every provider inspects the freshly parsed table and returns internal Actions. The client’s echoed diagnostics are used for linkage only (connecting an action to its squiggle in the UI) — never as input: providers recompute applicability from the table, so a client that drops or reorders diagnostic payloads changes nothing.
  4. Each text action ships a complete WorkspaceEdit in the changes-map form. The client applies the edit locally and follows up with an ordinary didChange — there is no second server round trip, which is why the server advertises resolveProvider: false and computes edits eagerly (they are cheap for CSV).

Walkthrough: the two dialect features

Reinterpret as … must change server state, not text — so it is the one action carrying an LSP Command instead of an edit. Picking it makes the client send workspace/executeCommand with csv-lsp.setDialect [uri, "ssv"]; the handler parses the arguments defensively (they arrive as loose JSON values — shape mismatches answer InvalidParams, never panic), re-parses the document under the new dialect, and republishes diagnostics. The response itself is null: effects travel as notifications.

Convert to … is a text edit with a twist — after the client applies it, the server must parse the new text under the new dialect, but it cannot know whether you picked the action or dismissed the menu. The trick lives in one field of the server state:

/// Dialect conversions offered in the last `codeAction` response, per URI:
/// `(converted full text, target dialect)`. When a `didChange` carries
/// exactly one of these texts, the user applied that conversion and the
/// document flips dialect. Cleared on every change/close, so dismissed
/// actions cost nothing.
pending_transforms: HashMap<String, Vec<(String, Dialect)>>,

On every didChange the server takes the entry for that URI and compares the incoming text against each recorded conversion; an exact match adopts that dialect for the reparse. A manual edit that happens to produce the exact conversion is the conversion — and any other edit clears the record.

Positions and encodings — the genuinely tricky bit

LSP positions are (line, character), where character counts code units of the negotiated encoding — not bytes, not codepoints. The historical default is UTF-16 (VS Code’s internal string encoding). In the line 😀x, the column of x is:

encodingcolumn of xbecause
utf-8 (bytes)4😀 is 4 UTF-8 bytes
utf-162😀 is a surrogate pair (2 units)
utf-32 (codepoints)1😀 is 1 codepoint

Since LSP 3.17 the client may offer general.positionEncodings in initialize; the server picks one and echoes it in its capabilities. csv-lsp prefers utf-8 (internal offsets are already bytes — conversion becomes trivial; Helix offers it), then utf-32, and falls back to the mandatory utf-16:

if offered.contains(&PositionEncodingKind::UTF8) {
    PositionEncoding::Utf8
} else if offered.contains(&PositionEncodingKind::UTF32) {
    PositionEncoding::Utf32
} else {
    PositionEncoding::Utf16
}

All three are implemented in position.rs — the only module allowed to convert. LineIndex records the byte offset of every line start (splitting on \n, \r\n and lone \r, exactly as the LSP spec defines line breaks) and converts in both directions, with the spec’s clamping rules for hostile input:

  • a line past the end of the document maps to the document end;
  • a character past the end of its line clamps to the line’s content end (before the terminator);
  • a character landing inside a multi-unit character — between UTF-16 surrogate halves, say — snaps back to that character’s start, so the returned offset is always a valid char boundary.
The CRLF corner case

One char boundary exists that no LSP position can express: the point between \r and \n of a CRLF break (characters clamp to the content end, which lies before the \r). Any edit with a boundary there would be misapplied by a conforming client. edits::minimize therefore widens its diffs off CRLF middles, and the property suite checks every span the server emits for client-addressability — the round trip offset → position → offset must be the identity. See the invariant map.

Rust concept · partition_point

Offset → line is a binary search over the sorted line starts: self.line_starts.partition_point(|&start| start <= offset) - 1. partition_point returns the index of the first element for which the predicate turns false — the standard library’s idiomatic “find my bucket” tool (also used by Table::row_at/cell_at to resolve the cursor). The - 1 is safe because index 0 is always present: line 0 starts at byte 0.

What the server advertises, and why

A server must never advertise what it cannot answer — clients are entitled to call anything claimed here (capabilities.rs):

CapabilityValueWhy
positionEncodingnegotiatedutf-8 preferred; see above
textDocumentSyncFULL + open/closewhole text on every change; reparse is O(n) and cheap, delta bookkeeping is not
codeActionProviderkinds quickfix, source, source.fixAll, refactor, refactor.rewrite; resolveProvider: falseedits are computed eagerly — no lazy resolve round trip
documentFormattingProvidertrue:format = align columns; idempotent by construction
documentHighlightProvidertruethe cursor’s column as per-cell ranges — Helix’s Spaceh turns them into a column multi-selection
executeCommandProvidercsv-lsp.setDialectthe one action that changes server state instead of text

Kind strings are hierarchical: a client filtering with only: ["source"] must also receive source.fixAll actions. The registry implements the spec’s dotted-prefix matching once, centrally — source matches source.fixAll, but quickfix does not match a hypothetical quickfixes.

Hardening: errors, panics, hostile clients

The failure-handling rules follow the two message flavors:

  • Requests must always be answered. Malformed params become an InvalidParams error response; unknown methods become MethodNotFound; and every handler runs inside catch_unwind, so a bug in one feature answers that one request with InternalError instead of killing the server.
  • Notifications are never answered — malformed ones are logged and dropped. Unknown documents in a didChange are tolerated (a client protocol error the server shrugs off), as is a ranged change under FULL sync.
  • Unknown state is not an error. A request about a never-opened URI returns an empty result; documentHighlight off any cell returns None — clients call these speculatively.

This is not defensive guesswork: the hostile-client property suite generates thousands of adversarial sessions — reversed ranges, positions past EOF or inside surrogate pairs, junk command arguments, unknown methods — and asserts the server answers everything and shuts down cleanly.

Rust concept · catch_unwind as a boundary, not a habit

Rust panics unwind the stack like exceptions but are not control flow — the idiom is to return Result everywhere and reserve std::panic::catch_unwind for process boundaries where “this thread must survive arbitrary bugs” is a requirement (the same pattern web servers use per-request). The state is wrapped in AssertUnwindSafe: the compiler cannot prove a &mut ServerState stays consistent across an unwind, so it makes you say it out loud. That promise is honest here because handlers mutate state only in ways that stay coherent if abandoned (worst case: a stale pending-transform offer, cleared on the next change).

Logging discipline

stdout belongs to the protocol — one stray byte corrupts the frame stream. All human-facing output goes to stderr via the log! macro, disabled unless CSV_LSP_LOG=1 is set. Helix surfaces the server’s stderr in its own log when started with hx -v. The stdio smoke test (tests/stdio.rs) exists precisely to catch stdout pollution forever: it spawns the real binary and fails if a single non-protocol byte appears on the pipe.
