csv-lsp documentation

A language server for CSV, TSV, SSV and PSV files — built for Helix, working with any LSP client. These pages document how to use it, how it works inside, and the Rust it is built from.

Editing delimiter-separated files by hand is fiddly: a missing cell breaks the column contract three screens later, quoting rules are easy to violate, and nothing lines up. csv-lsp reparses the file on every keystroke with an error-tolerant parser and turns problems into precise editor diagnostics — with fixes, formatting, and column-level refactors attached.

Orientation in ten lines

  • One binary, one thread. main.rs is a stdio shim; everything else is a library (src/lib.rs) so tests can drive the real server in-memory.
  • All internal offsets are byte offsets. Conversion to LSP (line, character) positions happens in exactly one place (position.rs), in the encoding negotiated with the client.
  • The parser is total. Any input produces a Table plus recovered ParseErrors — never a failure, never a panic.
  • Features are plugins over the parsed table. A feature is one module in src/features/ plus one line in Registry::standard().
  • Four runtime dependencies (lsp-server, lsp-types, serde_json, unicode-width) and one dev-dependency (proptest). Every one of them is a documented decision.

About these pages

The pages are plain static HTML with one shared stylesheet (style.css) and one small script (docs.js) — no build step, no external requests. They follow your system’s light/dark preference by default; the toggle in the header overrides it, remembered per browser. Read them hosted at tammoippen.github.io/csv-lsp (deployed from docs/ by GitHub Pages on every push to main), or open docs/index.html straight from a clone — everything loads relatively, so both work identically.

The feature GIFs are the one optional extra: they live in docs/gifs/ and are recorded (not committed until you record them) with the scripts in docs/gifs/tapes/ — see Recording the GIFs. Pages degrade gracefully while a GIF is missing.

Two other sources of documentation complement these pages:

  • The README — the user-facing summary, including the full Helix languages.toml and the rainbow-grammar setup.
  • The API docs — every public item carries a doc comment (enforced by the missing_docs lint); build them with cargo doc --no-deps --open.
