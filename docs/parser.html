The CSV parser

The project’s core asset: a hand-written, error-tolerant state machine that turns any byte sequence into a span-rich table — never failing, never panicking, never losing layout.

Why hand-written?

BurntSushi’s csv/csv-core is the canonical Rust CSV reader — and the wrong tool here. An LSP server is an IDE backend, and its parser has different requirements than a data reader, for the same reason rust-analyzer does not parse Rust with syn:

What the LSP needsWhat the csv crate offers
per-cell byte spans (cell vs content vs quotes vs padding) for diagnostics and precise editsper-record positions only; field spans not exposed
error reporting with recovery (unclosed quote, stray quote, text after closing quote)lenient by design: silently “repairs” malformed input — destroying exactly the information to surface
lossless layout model (alignment padding, original quoting, CRLF, BOM) so align ⇄ compact round-tripsnormalizes on read; layout is discarded
tolerate spaces after a closing quote ("abc" , — produced by the align feature itself)not modeled
multi-line quoted cells mapped back to editor linesspans missing

So parse.rs is ~400 lines of owned, deeply tested state machine — with csv-core kept only as a behavioral reference for edge cases. The consequence is embraced: the parser gets the deepest test coverage in the project, and every RFC 4180 edge-case decision is documented here instead of inherited silently from a library.

CSV in five minutes

RFC 4180 defines the format: records separated by line breaks, fields separated by ,. A field may be quoted with "…"; inside quotes, delimiters, line breaks and escaped quotes ("" → one literal ") are content. Which means this is one record with two fields:

"first
line",second

Consequence: row ≠ line. Everything in this crate therefore works on byte spans, and the line index maps spans to editor lines at the boundary — multi-line cells cost nothing extra.

The four dialects (CSV , · TSV tab · SSV ; · PSV |) differ only in the delimiter byte; quoting rules are identical. Real-world files deviate from the RFC constantly (5" bolt…), so the parser is total: it never rejects input, it reports deviations as recoverable errors with exact spans.

The data model

pub struct Span { pub start: usize, pub end: usize }   // byte offsets, half-open

pub struct Table {
    pub rows: Vec<Row>,                 // all rows, blank ones included
    pub errors: Vec<ParseError>,        // recovered problems, in source order
    pub dialect: Dialect,
    pub line_terminator: LineTerminator, // first terminator seen: Lf | CrLf
    pub has_bom: bool,                   // spans never include the BOM
    pub ends_with_newline: bool,         // no phantom empty row for it
}

pub struct Row { pub span: Span, pub cells: Vec<Cell> }  // span EXCLUDES the terminator

pub struct Cell {
    pub span: Span,           // full extent incl. padding, excl. delimiters
    pub content_span: Span,   // padding-trimmed; quoted cells INCLUDE the quotes
    pub quoting: Quoting,     // Unquoted | Quoted
    pub has_escaped_quotes: bool,   // "" occurred → value() must allocate
}

pub fn parse(text: &str, dialect: Dialect) -> Table;   // total: never fails

The key idea is the two spans per cell. Aligned files pad cells with spaces so delimiters line up; padding is layout, values are data, and the model keeps them apart without storing anything extra — padding is derived as span.start..content_span.start (leading) and content_span.end..span.end (trailing). For the row id, "a""b" ,x:

  • An all-padding cell gets a zero-width content span at span.start — the spaces count as trailing padding, consistent with left-aligned columns.
  • A blank row is exactly one unquoted cell with empty content. Blank rows are separators: legal, skipped by column checks, rendered as empty lines.
  • The header is the first non-blank row; its cell count is the column contract for the whole file (Table::expected_columns).
  • Span is Copy — two usizes move cheaper than a reference — and its constructor debug_assert!s start <= end, so inverted spans blow up in tests and cost nothing in release builds.
Rust concept · Cow — decode lazily, allocate rarely

Cell::value returns Cow<'t, str> (“clone on write”), an enum of Borrowed(&'t str) or Owned(String):

pub fn value<'t>(&self, text: &'t str) -> Cow<'t, str> {
    let content = self.content_span.slice(text);
    match self.quoting {
        Quoting::Unquoted => Cow::Borrowed(content),
        Quoting::Quoted => {
            let inner = content.strip_prefix('"').unwrap_or(content);
            let inner = inner.strip_suffix('"').unwrap_or(inner);
            if self.has_escaped_quotes {
                Cow::Owned(inner.replace("\"\"", "\""))
            } else {
                Cow::Borrowed(inner)
            }
        }
    }
}

Unquoted and cleanly-quoted cells borrow straight from the document text — the lifetime 't ties the result to the text, and the borrow checker guarantees no value outlives the document it points into. Only the "" escape forces an allocation, and the parser recorded whether one occurred so decoding never has to scan twice. Note the double unwrap_or: an unclosed cell has no closing quote to strip, so both strips are optional — recovery reaches all the way into decoding.

The state machine

The parser scans bytes, cell by cell. Each cell starts in CellStart, runs through at most one quoted or unquoted phase, and reports how it ended — on a delimiter (another cell follows in this row) or at a row end:

The same machine as a transition table:

StateInputEffect
CellStartspaceconsume — leading padding
"→ InQuoted
delim / EOL / EOFemit an empty cell
other→ InUnquoted
InUnquoteddelim / EOL / EOFemit — content = raw minus outer spaces
"record StrayQuote; the quote is literal content; stay
InQuoted""escaped quote — flag the cell; stay
"→ AfterQuoted (content ends after this quote)
CR / LF / othercontent — this is how cells span lines; stay
EOFrecord UnclosedQuote at the opening quote; emit
AfterQuotedspacetolerated silently — align’s own layout
delim / EOL / EOFemit
otherrecord TextAfterClosingQuote over the garbage run (trailing spaces trimmed); skip; the quoted value is kept

After each cell: a delimiter continues the row; \n, \r\n or lone \r ends it (the row span excludes the terminator; the file’s first terminator is recorded for re-rendering); EOF ends it with ends_with_newline = false. In code, the unquoted scan is a loop that breaks with the reason the cell ended:

let end = loop {
    match self.bytes.get(self.pos) {
        None | Some(b'\n' | b'\r') => break CellEnd::RowEnd,
        Some(&b) if b == self.delimiter => break CellEnd::Delimiter,
        Some(b' ') => self.pos += 1, // padding, unless content follows
        Some(b'"') => {
            // Stray quote: report it, keep it as literal content.
            self.error(ParseErrorKind::StrayQuote, self.pos, self.pos + 1);
            self.pos += 1;
            content_end = self.pos;
        }
        Some(_) => {
            self.pos += 1;
            content_end = self.pos;
        }
    }
};
The safety property that makes byte-wise scanning legal

The scanner walks &[u8], yet all spans must lie on char boundaries of the &str. That holds by a property of UTF-8: every byte of a multi-byte sequence has the high bit set, so the ASCII bytes the parser splits on (delimiter, quote, CR, LF, space) can never occur inside a multi-byte character. Splitting only at those bytes means splitting only at character boundaries — no char_indices() needed, no O(n) decoding per lookup, and the property suite still verifies is_char_boundary for every span the parser ever produces.

Rust concept · loops that return values, matches that see patterns

Two idioms carry this parser. break with a value turns a scan loop into an expression (let end = loop { … break CellEnd::Delimiter … }) — no mutable “reason” flag. And slice access via self.bytes.get(self.pos) yields an Option<&u8>, so one match handles EOF (None) and byte classes (or-patterns like Some(b'\n' | b'\r'), and a guard for the run-time delimiter) in a single exhaustive block: the compiler forces every input class to be handled — there is no “default: fall through” to forget.

Error recovery

Three kinds of quoting damage are recovered, each with a deliberate span policy — the span is what the user sees underlined:

KindExampleError spanRecoverySeverity
UnclosedQuote"abc … EOFthe opening " (1 byte)the rest of the file becomes the cell’s contenterror
StrayQuote5" boltthe stray "quote is literal content, keep scanningwarning — extremely common in the wild
TextAfterClosingQuote"x" y,zthe garbage run (y)garbage skipped; the cell’s value stays xerror

Every error records the index of the row it sits on, and that index powers the cascade prevention rule used across the codebase: rows overlapping a parse error are excluded from column-count checks (one unclosed quote must not produce fifty ragged-row errors) and are passed through verbatim by every rewrite — the renderer never reformats what the parser could not fully understand.

Gotcha worth internalizing

ends_with_newline must come from the parser’s terminator handling, not from text.ends_with('\n') — a trailing newline inside an unclosed quote is cell content, not a row terminator. This is the kind of interaction that makes the recovery rules load-bearing for every feature downstream.

The invariant catalog

These are the structural guarantees everything downstream (rendering, edits, features) builds on. They are machine-checked: assert_table_sound in tests/properties.rs asserts every one of them over thousands of generated documents (the invariant map shows which property covers what).

  1. Spans nest: content_span ⊆ span ⊆ row.span, and every offset lies on a char boundary of the text.
  2. Rows tile the text: adjacent rows are separated by exactly one terminator; the last row ends at EOF, or one terminator before it iff ends_with_newline. Nothing is lost, nothing is invented.
  3. Cells tile their row: the first cell starts at the row start, the last ends at the row end, and neighbors straddle exactly one delimiter byte.
  4. Padding is spaces: leading padding is always ASCII spaces; trailing padding is too on clean rows (on a text-after-quote row the trailing range holds the garbage bytes — and such rows are never reformatted).
  5. Blank ⇔ all-space: a row is blank exactly when its text is empty or spaces only.
  6. Errors sit on their rows: error.row is in range and the error span lies within that row’s span.
  7. Decoding is total: cell.value(text) never panics for any table the parser can produce.

Cursor resolution

Features constantly ask “which cell is the cursor on?”. Table::row_at and cell_at answer with binary search (partition_point over span starts) and two deliberate conventions, chosen to match what feels right in an editor:

  • Row ends are inclusive: a cursor sitting at the end of a row — on its terminator — still belongs to that row, so end-of-line quickfixes work.
  • A cursor on a delimiter resolves to the cell left of it; a cursor at the row end to the last cell.

Both helpers are total for any offset (in bounds or not), and they work inside multi-line quoted cells — an offset on the second editor line of a quoted cell still resolves to the row that owns it.

How the parser is tested

Four layers, each catching what the previous cannot (the testing page has the full story):

  1. Span-slice unit tests. Assertions compare text, not indices: assert_eq!(cell.span.slice(text), " bb ") — failures read like CSV, not like off-by-one arithmetic. Every state-machine arm and every recovery rule has a named test (unclosed_quote_swallows_the_rest_and_reports_the_opening_quote).
  2. The corpus. ~25 adversarial snippets (""", \r\r\n, BOM-only, emoji straddling quotes, \0 bytes…) × all four dialects, asserting only totality: no panic, spans in bounds and on char boundaries, decoding never fails.
  3. Property tests. proptest generates thousands of CSV-shaped documents and asserts the entire invariant catalog plus the render round-trips (align/compact idempotence, value preservation, lossless dialect conversion) — the inputs nobody thought to hand-pick.
  4. Golden renders. The renderer’s output for representative inputs is compared against inline string literals — layout decisions (BOM, CRLF, final newline, verbatim error rows) stay pinned down exactly.
